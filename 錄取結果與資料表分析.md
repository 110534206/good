# 錄取結果頁面與相關資料表分析

## 一、班導／指導老師「查看錄取結果」頁面

- **頁面**：`good/frontend/templates/user_shared/admission_results.html`
- **API**：`GET /admission/api/get_all_admissions`（實作在 `good/backend/admission.py`）

### 目前資料來源

API 的查詢是：

```sql
FROM teacher_student_relations tsr
LEFT JOIN semesters s ON s.id = tsr.semester_id
JOIN users u_student ON tsr.student_id = u_student.id
LEFT JOIN classes c ON u_student.class_id = c.id
LEFT JOIN student_preferences sp ON tsr.student_id = sp.student_id   -- 僅用 student_id
LEFT JOIN internship_companies ic ON sp.company_id = ic.id
LEFT JOIN internship_jobs ij ON sp.job_id = ij.id
LEFT JOIN users u_teacher ON tsr.teacher_id = u_teacher.id
```

也就是說：

- 以 **teacher_student_relations** 為基底（每一筆 = 某學期、某指導老師、某學生）。
- **LEFT JOIN student_preferences** 只依 `tsr.student_id = sp.student_id`，沒有再限制學期或公司。

因此：

- 一位學生若有多筆 `teacher_student_relations`（多位指導老師）且有多筆 `student_preferences`（多個志願／公司），就會出現 **多筆列**。
- 班導／指導老師畫面上就會看到「同一學生、多筆錄取結果」。

## 二、五張資料表與流程對應

| 資料表 | 用途簡述 |
|--------|----------|
| **student_job_applications** | 學生投遞履歷紀錄（學生–公司–職缺）。一筆 = 一次投遞。 |
| **resume_teacher** | 指導老師審核學生履歷。`application_id` → `student_job_applications.id`，審核通過後履歷才給廠商。 |
| **resume_applications** | 廠商對該筆投遞的審核／面試／錄取排序。`application_id` → `student_job_applications.id`。欄位含 `apply_status`, `interview_*`, `is_reserve`, `slot_index`（媒合順序）。 |
| **teacher_student_relations** | 指導老師與學生的關聯（見下一節）。 |
| **users** | 使用者（學生、老師、班導、主任、科助、廠商等）。 |

流程對應：

1. 學生投遞 → **student_job_applications**
2. 指導老師審核（學生投遞的是該老師的實習公司）→ **resume_teacher**
3. 通過的履歷給廠商 → 廠商在 **vendor_review_resume** 頁做媒合排序 → 寫入 **resume_applications**（`slot_index`, `is_reserve` 等）
4. 主任在 **manage_director** 頁調整排序／確認 → **manage_director**
5. 科助在 **final_results** 公告 → 寫入 **internship_offers**（每位學生一筆最終錄取）

## 三、媒合結果、未錄取名單是否有專用資料表？

### 媒合結果

- **沒有**一張「媒合結果」專用表。
- 媒合結果是**多張表一起算出來的**：
  - 廠商排序：**resume_applications**（`is_reserve`, `slot_index`, `apply_status` 等）
  - 主任調整／決策：**manage_director**（`director_decision` = Approved/Rejected 等）
  - 志願與學期：**student_preferences**（`status`, `semester_id` 等）
  - 與投遞的對應：**student_job_applications**（`preference_id` 常對應 `student_job_applications.id`）
- **最終公告的結果**會寫入 **internship_offers**：每位學生一筆（`student_id`, `job_id`, `status`），即「正式錄取」的單一紀錄。

### 未錄取名單

- **沒有**「未錄取名單」專用表。
- 未錄取是**用查詢算出來的**：
  - 當前學期應實習學生（依班級／學期）
  - 扣掉「已在媒合結果中且為正取」的學生（程式裡是從 **resume_applications** + **student_preferences** + **manage_director** 等 JOIN 判斷誰算已錄取）。

## 四、teacher_student_relations 的儲存邏輯

### 表結構（依你提供的 SQL）

- `id`, `teacher_id`, `student_id`, `role`, `semester_id`, `created_at`
- 註：部分環境可能有 `company_id`、`semester`（字串）等欄位，依實際 schema 為準。

### 何時寫入／更新

1. **科助確認媒合結果**（`ta_confirm_matching`）時：
   - 針對每個「已核准」的媒合紀錄，取該公司的 `advisor_user_id`（指導老師）。
   - 若不存在該 (老師, 學生, 學期) 的 relation，則 **INSERT** 一筆 `teacher_student_relations`（role 如「指導老師」）。
   - 若已存在則可能 **UPDATE**（例如 `updated_at` 或若有 `company_id` 則更新）。

2. **廠商確認錄取**（vendor 流程中的 `_record_admission_and_bind_relation`）時，也可能寫入或更新 **teacher_student_relations**，讓該學生與該公司指導老師產生關聯。

3. 若系統有「學期初分配指導老師」的功能，也可能在該處寫入（同一學生、多個老師、同一學期會有多筆）。

### 儲存邏輯整理

- 一筆 **teacher_student_relations** = 「在某一學期，某位老師是某位學生的指導老師」。
- 同一學生、同一學期可以有多位指導老師（多筆 relation），例如：
  - 不同公司對應不同指導老師；
  - 或班級／科系事先分配的多位指導老師。
- 因此 **不是**「一個學生只對應一個老師」，而是「(老師, 學生, 學期)」的關聯表。

## 五、為何班導／指導老師會看到「多筆」錄取結果？

- **真實情況**：每位學生的**最終**錄取只有一筆，存在 **internship_offers**（一學生一 job）。
- **目前 API**：  
  - 以 **teacher_student_relations** 為基底，再 **LEFT JOIN student_preferences ON student_id**（沒有再限學期或公司），所以會得到：
    - 每個 (老師, 學生, 學期) × 該學生**所有**志願（所有公司／職缺）。
  - 因此同一學生會出現多列（每列對應一個志願或一組 relation×preference），看起來就像「多筆錄取結果」。

也就是說：**畫面上的多筆是「該生的所有志願／關聯」被展開，而不是只有「最後錄取的那一筆」**。

## 六、建議修正方向（讓畫面只顯示「一筆錄取」）

若要讓班導／指導老師的「查看錄取結果」與 **internship_offers** 一致（一學生一筆），可以考慮：

### 方案 A：以 internship_offers 為單一真相來源

- **get_all_admissions** 改為以 **internship_offers** 為主資料源（再 JOIN 學生、班級、公司、職缺、學期）。
- 班導：篩選條件為「學生的 `class_id` 在我管理的班級」。
- 指導老師：篩選條件為「該錄取公司之 `advisor_user_id` = 當前老師」或「存在 `teacher_student_relations` 且 `teacher_id` = 當前老師，且該錄取公司為該老師所管」。
- 優點：畫面上每一學生只會有一筆，與 **internship_offers** 一致。

### 方案 B：維持現有資料源，但只顯示「最終錄取」的那一筆

- 仍用 **teacher_student_relations** + **student_preferences**，但：
  - 再 JOIN **internship_offers**（`student_id`, `job_id`），並限定只顯示「`internship_offers` 有紀錄」的那一筆 preference（即該 job 的志願）。
  - 或先查 **internship_offers** 得到 (student_id, job_id)，再只取對應的那一筆 preference/relation 來顯示。
- 優點：不需大改既有 JOIN 結構，只收斂成「有在 internship_offers 的那一筆」。

若你希望，我可以依你選的方案（A 或 B）直接改 `get_all_admissions` 的 SQL 與篩選邏輯，並對應調整前端顯示（例如不再需要「多筆同一學生」的合併邏輯）。

## 七、表關聯簡圖（供對照）

```
student_job_applications (學生投遞)
    ├── resume_teacher (指導老師審核) application_id → sja.id
    ├── resume_applications (廠商審核/排序/正備取) application_id → sja.id
    └── student_preferences (志願/學期/狀態) 透過 sja.student_id, company_id, job_id 等與流程串接

teacher_student_relations (老師–學生–學期)
    └── 用於：誰可以審核該生、班導/指導老師查看錄取時篩選範圍

internship_offers (最終錄取，一學生一筆)
    └── student_id, job_id, status, offered_at, responded_at
```

以上說明可作為你調整「查看錄取結果」與未來新增「媒合結果／未錄取」報表時的依據。

## 八、錄取結果／未錄取名單是否需要新增資料表？

### 結論摘要

| 項目         | 是否必須新增表 | 建議 |
|--------------|----------------|------|
| **錄取結果** | 否             | 現有 `internship_offers` 已可作為單一真相；若需區分學期或多學期並存，可加欄位或一張快照表。 |
| **未錄取名單** | 否           | 用查詢即時計算即可；若有「二輪媒合／補分發」或需留底，可選配一張快照表。 |

---

### 錄取結果

**現狀**  
- 最終錄取已寫入 **internship_offers**（一學生一筆：`student_id`, `job_id`, `status`, `offered_at`, `responded_at`）。  
- 沒有「學期」欄位，若未來同學生會有多學期錄取，無法單靠本表區分。

**選項（依需求擇一或不做）**

1. **不新增表，僅加欄位（推薦，若只跑單學期或可接受一表多學期）**  
   - 在 **internship_offers** 加 `semester_id`（FK → semesters）。  
   - 科助／主任確認或寫入時一併寫入當前學期。  
   - 優點：一表搞定、查詢簡單、一學生一學期一筆。  

2. **不新增表、也不加欄位**  
   - 維持現狀，學期用 **student_preferences** 等 JOIN 推斷（目前 get_all_admissions 做法）。  
   - 適合：只關心「最新一筆錄取」、不區分學期。  

3. **新增「公告快照」表（僅在需要正式公告留底時）**  
   - 例如：`admission_announcements`（`id`, `semester_id`, `student_id`, `job_id`, `company_id`, `announced_at`, `published_by`）。  
   - 科助「公告媒合結果」時寫入一筆，不覆蓋 `internship_offers`。  
   - 優點：可查「某學期當時公告了誰」、稽核；缺點：多一張表與寫入邏輯。

**建議**：若系統會處理多學期（例如 1132、1142 並存），優先做 **選項 1（internship_offers 加 semester_id）**；若僅單學期且不需公告留底，維持現狀即可，不必為「錄取結果」新增表。

---

### 未錄取名單

**現狀**  
- 未錄取 = 「當學期應實習學生」−「當學期已錄取（正取）學生」，由查詢即時算，沒有專用表。

**選項**

1. **不新增表（推薦）**  
   - 繼續用查詢：依學期篩選應實習學生，再排除已在 `internship_offers`（或依學期在 `student_preferences`／resume 流程）的已錄取者。  
   - 優點：永遠與現有錄取狀態一致、無同步問題。  

2. **新增「未錄取快照」表（選配）**  
   - 例如：`semester_unadmitted`（`id`, `semester_id`, `student_id`, `snapshot_at`）。  
   - 在「科助公告媒合結果」或「開放二輪媒合」時，寫入當下未錄取名單。  
   - 適用情境：二輪媒合／補分發只允許「該學期公告時未錄取」的學生、或要報表／留底。  
   - 缺點：需在公告或流程節點維護寫入；若之後有補錄取，要決定是否更新快照或僅供當時查詢。

**建議**：一般情況 **不新增表** 即可；只有當二輪媒合或報表明確需要「當次公告的未錄取名單」時，再考慮 **選項 2**。

---

### 若決定新增或調整時的建議欄位

- **internship_offers 加學期（建議）**  
  - `semester_id` INT NULL，FK → `semesters.id`；寫入時由當前流程學期帶入。  

- **admission_announcements（選配）**  
  - `id`, `semester_id`, `student_id`, `job_id`, `company_id`, `announced_at`, `published_by`（user_id）。  

- **semester_unadmitted（選配）**  
  - `id`, `semester_id`, `student_id`, `snapshot_at`；可再加 `source`（e.g. 'first_round' / 'second_round'）若日後要區分輪次。
