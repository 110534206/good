# 學生投遞履歷邏輯整理

## 一、流程總覽

```
學生投遞履歷 → 班導可查看（僅下載/查看，不審核）→ 履歷截止後自動視為班導通過
    → 同步到 resume_teacher（指導老師待審）→ 指導老師審核通過
    → 寫入 resume_applications → 廠商可見並審核
```

---

## 二、相關資料表

| 資料表 | 用途 |
|--------|------|
| **resumes** | 學生上傳的履歷檔案；`category='ready'` 才能投遞。審核狀態現以「每筆投遞」為單位，不在此表改為「審核中」。 |
| **student_job_applications** | 每筆投遞紀錄（學生＋公司＋職缺＋履歷）。唯一鍵：`(student_id, job_id)`，**同一學生同一職缺只能一筆**。 |
| **resume_teacher** | 每筆投遞的「班導／指導老師」審核狀態（以 `application_id`＋`teacher_id` 為單位）。 |
| **resume_applications** | 指導老師通過後才會有記錄；廠商依此表顯示履歷、填寫面試與審核狀態。 |

---

## 三、學生端操作

### 3.1 投遞履歷

- **API**：`POST /api/student/apply_company`
- **參數**：`company_id`, `job_id`, `resume_id`
- **條件**：
  - 已登入且角色為學生
  - 在「實習流程學期」內（可投遞期間）
  - 履歷為本人且 `category='ready'`（正式版本）
  - 公司、職缺存在且公司已審核通過、職缺啟用
- **限制**：同一學生、同一職缺只能一筆（`student_job_applications` 唯一鍵 `uniq_student_job`）。若已投過該公司該職缺，會回 400 並提示先刪除原投遞再重新投遞。
- **寫入**：僅新增一筆 `student_job_applications`（`status='submitted'`），**不**在此時建立 `resume_teacher` 或 `resume_applications`。

### 3.2 查看我的投遞

- **API**：`GET /api/student/my_applications`
- **說明**：列出該學生所有投遞；每筆的審核狀態以「每筆投遞」為單位，由 `resume_teacher`（指導老師 → 班導）與 `resumes` 彙總，不受「同一履歷投別間公司」影響。截止後仍可查看歷史。

### 3.3 刪除投遞

- **API**：`POST /api/student/delete_application`（body：`application_id`）
- **條件**：僅能刪除自己的投遞，且**須在履歷繳交截止時間之前**。
- **效果**：依序刪除 `resume_applications` → `resume_teacher` → `student_job_applications`，主任／班導／指導老師端該筆會一併消失。

---

## 四、班導／主任

- **班導**：僅能**下載、查看**學生履歷與留言，**不能**退件或通過（與主任設定一致）。可單選／全選列勾選（UI 與主任相同）。
- **主任**：僅能下載、查看。
- **資料來源**：班導／主任列表由 `GET /api/teacher_review_resumes` 取得，資料來自 `student_job_applications`＋`resumes`＋`resume_teacher` 等，顯示「投遞時間」等欄位。

---

## 五、履歷上傳截止時間後（自動視為班導通過）

- **觸發**：呼叫 `get_teacher_review_resumes` 等 API 時，會執行 `update_resume_status_after_deadline()`。
- **行為**：
  1. 將 `resumes` 表中 `status='uploaded'` 的履歷改為 `status='approved'`（視為班導通過）。
  2. 對所有「該履歷已 approved」的 `student_job_applications`，為對應公司的**指導老師**（`advisor_user_id`）在 `resume_teacher` 建立或同步一筆，`review_status='uploaded'`，表示待指導老師審核。
- **結果**：未退件的投遞會自動進入「等待指導老師審核」狀態。

---

## 六、指導老師審核

- **API**：`POST /api/review_resume/<resume_id>`（body 需帶 `application_id`、`status`：`approved` 或 `rejected`）。
- **寫入**：更新或新增 `resume_teacher` 中該 `application_id`＋該老師的 `review_status`、`comment`。
- **當 status = 通過（approved）時**：
  - 立即呼叫 `ensure_resume_application_for_teacher_approved()`：
    - 若尚無對應的 `resume_applications`（依 `application_id`＋`job_id`），則新增一筆（`apply_status='uploaded'` 等），廠商即可看到該履歷。
- **指導老師審核截止時間後**：`update_resume_applications_after_advisor_deadline()` 會對「指導老師已通過但尚未有 `resume_applications`」的投遞補建一筆，並可發通知給廠商。

---

## 七、廠商端

- **資料來源**：廠商看到的履歷以 **resume_applications** 為準；只有在此表有記錄的投遞才會顯示。
- **狀態**：`apply_status`（待審核／通過／退件）、面試狀態、廠商留言等皆存在 `resume_applications`。

---

## 八、時序與狀態對照

| 時點 | student_job_applications | resume_teacher | resume_applications |
|------|--------------------------|----------------|---------------------|
| 學生投遞 | 新增一筆 | 無 | 無 |
| 班導／主任查看 | 有 | 可能無或已有（若班導曾審核） | 無 |
| 履歷截止後 | 有 | 為指導老師建立/同步「待審」 | 無 |
| 指導老師通過 | 有 | 該老師記錄 = approved | **新增一筆**（立即） |
| 指導老師審核截止後 | 有 | 有 | 補建尚未有的記錄 |

---

## 九、重要程式位置（backend）

- 學生投遞：`company.py` → `apply_company()`
- 學生我的投遞：`company.py` → `get_my_applications()`
- 學生刪除投遞：`company.py` → `delete_application()`
- 履歷截止後自動通過＋同步 resume_teacher：`resume.py` → `update_resume_status_after_deadline()`
- 指導老師審核：`resume.py` → `review_resume()`
- 指導老師通過後立即建 resume_applications：`resume.py` → `ensure_resume_application_for_teacher_approved()`
- 指導老師審核截止後補建 resume_applications：`resume.py` → `update_resume_applications_after_advisor_deadline()`
- 主任／班導／指導老師列表：`resume.py` → `get_teacher_review_resumes()`
- 廠商履歷列表：`vendor.py` → `get_vendor_resumes()`（依 `resume_applications` 為主）

---

## 十、前端頁面

- 學生查看公司與投遞：`look_company.html`（投遞按鈕、我的投遞狀態、刪除按鈕）
- 班導審核（僅查看）：`class_review_resume.html`
- 主任／指導老師審核：`review_resume.html`（主任/班導僅下載與查看；指導老師可通過/退件）
