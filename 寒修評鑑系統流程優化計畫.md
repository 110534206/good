# 寒修評鑑系統流程優化計畫

## 核心目標
優化寒修評鑑流程，包含二面機制、權限調整、審核功能簡化以及時間限制設定。

## 1. 二面機制 (Second Interview)
### 邏輯規劃
- **二面觸發**：一面未錄取之學生，科助可手動指派至二面單位。
- **面試輪次**：在 `resume_applications` 與 `interview_schedules` 增加 `round` 欄位 (1: 一面, 2: 二面)。
- **薪資差異處理**：
  - 在 `resume_applications` 增加 `offered_salary` 欄位。
  - 若 `offered_salary` 有值，則優先於職缺資訊中的 `salary`。
  - 二面指派時，系統自動帶入該單位的二面薪資設定（或由科助編輯）。

## 2. 履歷審核流程優化 (Resume Review)
### 職能調整
- **主任、班導**：改為「僅查看」權限，移除審核（通過/退件）按鈕。
- **系統自動轉交**：繳交期限截止後，履歷狀態自動由「已繳交」轉為「待指導老師審核」。
- **指導老師審核簡化**：
  - 移除「退件」功能，僅保留「通過」選項。
  - 審核介面增加「全選」與「勾選多筆批次送出」按鈕。
  - 指導老師僅需勾選欲推薦給廠商的學生，一併送出至廠商端。

### 2.1 現況：履歷／志願序截止時間與學期脫鉤
- **目前做法**：科助在 **通知頁** (`notifications.html`) 手動「新增作業截止時間」→ 寫入 `announcement` 表（標題為 `[作業]上傳履歷截止時間` / `[作業]填寫志願序截止時間`），系統取「最新一筆」的 `end_time` 作為截止時間。
- **問題**：
  - 截止時間**沒有綁定學期**：換學期後若沒重新設公告，可能沿用舊學期或搞混。
  - **多學期並存**（如 110 屆 1132、111 屆 1142）時，無法區分「哪個截止屬於哪個學期」。
- **程式依賴**：
  - `announcement.py`：`/api/check_deadline?type=resume|preference`、`/api/save_deadlines` 皆依賴 `announcement` 表。
  - `resume.py`：`update_resume_status_after_deadline()` 從 `announcement` 取履歷截止時間，過期後將 `resumes.status` 由 `uploaded` 改為 `approved`，並同步到 `resume_teacher`（待指導老師審核）。

### 2.2 建議做法：以學期為主、公告為輔
| 項目 | 建議 |
|------|------|
| **資料來源** | 在 `semesters` 表新增 `resume_submit_deadline`、`pref_submit_deadline`（DATETIME），由**學期管理**設定「當前學期」的繳交截止。 |
| **讀取邏輯** | `check_deadline`、`update_resume_status_after_deadline` 改為：**優先讀取當前學期**（`is_active=1`）的上述欄位；若為 NULL，再 fallback 到現有 `announcement` 查詢（向後相容）。 |
| **科助操作** | 在**學期管理**頁設定該學期的「履歷繳交截止」「志願序繳交截止」；通知頁的「新增作業截止時間」可改為：寫入學期欄位並**順便發一筆公告/通知**（或改為僅顯示學期設定，不再獨立寫 announcement）。 |
| **履歷審核觸發** | 「繳交期限截止後自動轉交指導老師」的依據改為**當前學期的 `resume_submit_deadline`**，與學期切換一致。 |

### 2.3 操作流程建議（科助）
1. **學期切換前**：在學期管理為新學期設定 `resume_submit_deadline`、`pref_submit_deadline`（及既有規劃的 `advisor_review_deadline`、`vendor_review_deadline`）。
2. **需要提醒學生時**：可在通知頁「發布公告」提醒截止日，或由系統依學期截止日自動產生一筆提醒公告（可選實作）。
3. **不再依賴**：僅靠「在通知頁手動新增一筆截止時間公告」作為唯一依據，避免與學期不同步。

## 3. 廠商審核簡化 (Vendor Review)
- **按鈕功能調整**：僅保留「通知已通過 (錄取)」與「不通過 (未錄取)」兩個選項。
- **通知邏輯**：點選後自動發送系統內通知或 Email 給學生。

## 4. 審核時間限制 (Time Limit Settings)
- **參數設定**：在學期管理 (Semester) 中增加 `advisor_review_deadline` 與 `vendor_review_deadline`。
- **權限控制**：
  - 超過指導老師審核期限，指導老師審核功能自動失效，標為主動關閉。
  - 超過廠商審核期限，廠商審核功能失效，並提醒由科助介入處理。

## Proposed Changes

### [Component: Database]

#### [MODIFY] `semesters` (Table)
- 新增 `resume_submit_deadline` (DATETIME): 履歷繳交截止時間（學生上傳／班導端「已繳交」的截止）。
- 新增 `pref_submit_deadline` (DATETIME): 志願序繳交截止時間。
- 新增 `advisor_review_deadline` (DATETIME): 指導老師審核截止時間。
- 新增 `vendor_review_deadline` (DATETIME): 廠商審核截止時間。

#### [MODIFY] `resume_applications` (Table)
- 新增 `round` (INT, default 1): 面試輪次。
- 新增 `offered_salary` (DECIMAL): 該次應徵之合約薪資（二面可能不同）。

## Verification Plan

### Automated Tests
- 使用 `pytest` 撰寫單元測試驗證 `round` 與 `offered_salary` 的寫入與讀取。
- 撰寫 API 測試驗證超過 deadline 時是否正確回報 `403 Forbidden` 或 `400 Bad Request`。

### Manual Verification
1. 以科助帳號指派一名一面未錄取名單學生進入二面單位，確認薪資是否可變動。
2. 以指導老師帳號登入，確認是否可批次送出履歷。
3. 以主任帳號登入，確認審核按鈕是否已消失。
4. 調整伺服器時間（或資料庫時間）至截止後，確認老師與廠商端是否無法再進行操作。
