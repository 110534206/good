# 智慧實習系統 — 功能統整說明

> 本文件統整目前系統已具備的功能模組與頁面，供後續規劃細部新功能時參考。

---

## 一、系統架構概覽

- **技術**：Flask 後端，前後台共用同一 `app.py`，Jinja2 模板
- **前端**：前台 `frontend/`、後台 `admin_frontend/` 分離模板
- **API**：RESTful 風格，多數功能透過 `/api/*` 提供
- **排程**：APScheduler，每 60 分鐘檢查學期自動切換等任務
- **角色**：學生、指導老師、班導師、主任、科助(TA)、管理員、廠商

## 三、依模組區分之功能

### 3.1 認證 (auth)

- 登入、登出、角色確認、切換角色（班導/老師/主任）
- 學生註冊、廠商註冊
- 訪客頁、登入頁、註冊選擇頁

### 3.2 使用者與個人資料 (users)

- 各角色首頁：學生、老師、班導、主任、TA、廠商
- 個人資料頁與 API
- 班導專用：班導首頁、班級履歷審核、班級志願序相關

### 3.3 履歷 (resume)

- 履歷填寫、儲存、載入、產出 Word（submit_and_generate）
- 證照：證照機關、證照項目、證照照片上傳
- 成績：成績單上傳、專業核心科目、Excel 匯入成績、個人模板
- 缺曠：缺曠統計、學期、預設區間、紀錄查詢與提交
- 履歷審核：老師/班級審核、狀態更新、下載
- 履歷夾：`resume_folders` 頁面與相關 API（部分註解）
- 職缺類別與層級：`/api/get_job_categories_and_levels`
- 公司職缺列表：`/api/company_positions`

### 3.4 公司 (company)

- 公司上傳、審核、查看
- 實習單位資料 Word 產出、公司文件上傳 (docx/doc)
- 實習 Q&A（前台與後台管理）

### 3.5 志願序 (preferences)

- 學生填寫志願、班導/老師審核
- 截止時間後自動將 submitted 改為 approved
- 志願序匯出（Excel 等）、錄取結果相關

### 3.6 錄取與實習生管理 (admission)

- 錄取結果頁（老師、班導、TA、admin）
- 實習生管理頁（老師、主任、TA、admin、廠商）
- 記錄錄取 API：綁定指導老師與學生、寫入 internship_offers 等
- 取得我的錄取、取得實習生名單等 API

### 3.7 實習心得 (intern_exp)

- 實習心得列表與表單
- 依公司篩選、提交、審核
- 與 email、通知整合

### 3.8 公告 (announcement)

- 作業狀態檢查（是否在申請期間）
- 公告列表 API
- 公告管理、發布（含廠商發布）、時間區間與內容清理

### 3.9 通知 (notification)

- 站內通知中心頁
- 建立通知（依關鍵字自動分類：履歷、志願、心得、公司、審核等）
- 已讀/未讀、列表 API

### 3.10 學期 (semester)

- 當前學期、學期代碼、學期 ID（供其他模組使用）
- 學期列表、新增/編輯/刪除、設定當前學期
- 排程自動切換學期（每 60 分鐘）

### 3.11 主任總覽 (director_overview)

- 主任所屬科系
- 全系各班履歷進度與完成率

### 3.12 科助統計 (ta_statistics)

- 全系統總覽：學生數、履歷、志願序、公司統計
- 公司被選次數排行
- 報表匯出等（依實際 route 為準）

### 3.13 學生實習成果 (student_results)

- 學生「我的實習成果」頁
- API：取得錄取結果與實習心得

### 3.14 AI 輔助 (ai_tools)

- 使用 Google Gemini 潤飾履歷/自傳/申請訊息
- API：`/api/revise-resume` 等
- 語氣選項：專業正式、親切隨和等

### 3.15 廠商 (vendor)

- 職缺、應徵、面試、公告、通知、郵件紀錄、媒合排序（見 2.7）

### 3.16 後台管理 (admin)

- 用戶管理、學期、缺曠預設、實習 Q&A（見 2.6）

## 五、現有功能細部完善建議

以下針對**目前已有功能**提出「更完善的設置」構思，不新增大框架，而是把既有流程補齊、減少人工失誤、提升可維護性。

---

### 5.1 履歷相關（含必修／專業核心科目 Excel）

**現狀簡述**：TA 可匯入「專業核心科目」清單、TA 可匯入「學生成績 Excel」、學生可下載範本、學生可上傳「已修習專業核心科目」Excel；履歷表單內有成績欄位與成績單上傳。

| 完善項目 | 說明 | 實作方向建議 |
|----------|------|----------------|
| **範本與欄位說明** | 避免學生/TA 填錯欄位 | 提供「下載範本」時一併提供欄位說明（學號、課程名稱、學分、成績/等第）；範本內第一列或第二列可為說明列；上傳頁面顯示「建議欄位名稱」與範例。 |
| **匯入前預覽與錯誤回報** | 目前失敗時僅回傳訊息，不易除錯 | TA 匯入成績 Excel 時：先做**預覽**（前幾筆、欄位對應是否正確），再確認寫入；回傳時列出「成功 N 筆、跳過 M 筆、錯誤清單（行號+原因）」。 |
| **學號與系統用戶對應** | 學號可能打錯或非本系統學生 | 匯入前/匯入時檢查學號是否對應 `users` 表且 role=student；無法對應的列可列入「跳過清單」並回報，不靜默略過。 |
| **必修 vs 已修比對** | 學生是否缺修必修科目 | 以 `standard_courses`（TA 匯入的核心科目）為基準，比對學生 `course_grades`：在履歷頁或審核端顯示「缺修必修：OO 課程」，或標示「已修畢全部核心科目」。 |
| **成績格式與範圍驗證** | 成績/等第格式不一（數字、等第、空白） | 定義允許格式（如 0–100、A/B/C/或等第表）；匯入或儲存時驗證，不符時回傳明確錯誤（欄位名+行號），避免整批失敗。 |
| **覆蓋前確認** | TA 匯入成績會覆蓋該學期該生舊資料 | 匯入前提示「將覆蓋當前學期該批學生的既有成績」，可選「僅新增/更新、不刪除舊列」或「完整覆蓋」；必要時提供「匯入紀錄」供回溯。 |
| **學生端 Excel 與表單一致** | 學生上傳的科目 Excel 與表單手動輸入可能重複 | 上傳後自動帶入「已修習專業核心科目」區塊，並可與手動編輯並存或鎖定其一；若已有手動輸入，提示「將以 Excel 覆蓋」並可取消。 |
| **成績單圖片與成績欄** | 成績單圖片與 Excel/表單成績可能不一致 | 審核端可同時看到「成績單圖片」與「表單/Excel 成績」，方便比對；若有差異可標註「待確認」不擋審核。 |
| **缺曠與履歷關聯** | 缺曠資料已在系統，履歷產出是否帶入 | 產出 Word 時可選「是否含缺曠摘要」；缺曠預設區間與學期連動（學期切換時已部分存在，可檢查是否全流程一致）。 |
| **版本快照 (Resume Snapshot)** | 學生常改來改去改壞了，需要可還原的版本 | 增加「儲存當前版本」按鈕，可為版本命名（例如「投遞 A 公司」）；履歷列表或編輯頁提供「版本歷史」，可隨時還原到某個快照，避免覆蓋後無法復原。 |

### 5.2 學期系統設置

**現狀簡述**：學期 CRUD、當前學期、`auto_switch_at` 排程每 60 分鐘檢查並自動切換；切換時會更新缺曠預設區間。

| 完善項目 | 說明 | 實作方向建議 |
|----------|------|----------------|
| **切換前檢查（未完成事項）** | 避免學期一換，上學期資料未收尾 | 手動/自動切換前：檢查「當前學期」是否還有「未審核履歷／未審核志願序／待審公司」；若有，列出筆數並可選擇「仍要切換」或「取消」，並寫入日誌。 |
| **切換前摘要頁** | 管理員/TA 不知道切換後影響範圍 | 在「設定當前學期」或「預約自動切換」旁提供「切換影響摘要」：新學期代碼、將關閉的舊學期、缺曠區間會變更的屆別、建議先完成的作業（如審完志願）。 |
| **禁止刪除有關聯的學期** | 目前註解提到可加檢查，尚未實作 | 刪除學期前檢查：該學期是否有履歷、志願序、錄取、缺曠等；若有則禁止刪除並列出關聯表與筆數，或改為「停用/封存」不刪除。 |
| **學期與公告/作業時間** | 志願序截止等依公告時間，與學期可能脫鉤 | 學期管理頁顯示「當前學期」；若有「填寫志願序截止」等公告，可顯示其 end_time 與當前學期的對應關係，避免學期已換但公告仍寫舊學期。 |
| **自動切換日誌與測試** | 排程是否執行、是否切換成功不易確認 | 每次 `check_auto_switch` 執行時寫入日誌（時間、是否找到待切換、切換結果）；後台提供「最近自動切換紀錄」或「測試自動切換」按鈕（僅測試用學期）。 |
| **學期與資料範圍一致性** | 各模組取「當前學期」方式是否一致 | 統一由 `get_current_semester_id` / `get_current_semester_code` 取得；履歷、志願序、缺曠、錄取等寫入時都帶學期 ID/代碼，避免新學期還寫到舊學期。 |

### 5.3 媒合機制（志願序與錄取）

**現狀簡述**：學生填志願序、班導/老師審核；截止後可將 submitted 改為 approved；錄取由廠商/老師手動 `record_admission`，寫入 `internship_offers` 並綁定指導老師與學生。

| 完善項目 | 說明 | 實作方向建議 |
|----------|------|----------------|
| **錄取時帶入志願序** | 避免重複輸入、資料一致 | 廠商/老師在「記錄錄取」時，可先選學生，系統帶出該生「已審核通過的志願序」；選擇錄取對應的 company_id / job_id / preference_order，一鍵帶入並寫入 internship_offers + 更新該志願狀態。 |
| **志願序與錄取結果對應顯示** | 審核端與學生端都容易對照 | 錄取結果頁/實習生管理頁：顯示「錄取公司/職缺」與「該生第幾志願」；學生端「我的實習成果」可顯示「錄取為第 N 志願」。 |
| **未錄取／備取狀態** | 目前多為「已錄取」或無紀錄 | 若需區分：在 internship_offers 或志願序狀態支援「未錄取／備取／已錄取」；列表與匯出可篩選「尚未錄取」「備取」學生。 |
| **依志願序建議錄取名單** | 協助廠商/老師不遺漏 | 廠商查看應徵列表時，可依「學生志願序」排序（把該職缺填為第一志願的排前面）；或提供「建議錄取名單」：已將本職缺填為前 N 志願且尚未被錄取的學生。 |
| **志願序截止後鎖定** | 截止後不應再改志願 | 截止時間後：學生端僅能「檢視」不能編輯；若已有「submitted→approved」自動更新，可一併鎖定「已 approved」的志願不允許刪改（除非管理員還原）。 |
| **媒合結果匯出** | 方便存檔與統計 | 提供「本學期媒合結果」匯出：學生、志願序、錄取公司/職缺、指導老師、錄取時間；格式可為 Excel，欄位與現有錄取結果頁一致。 |
| **重複錄取檢查** | 一名學生不應被多間公司錄取（若規則為一人一實習） | record_admission 前檢查該生是否已有 status=accepted 的 internship_offers；若有則提示「該生已錄取 OO 公司」，可選擇覆蓋或取消。 |

### 5.5 履歷視覺化與 AI 輔助（構思）

以下為履歷模組的**進階構思**：視覺化分析、AI 關鍵字萃取與搜尋，可作為後續實作參考。

#### 5.5.1 視覺化分析 (Visualization & Analytics)

| 項目 | 說明 | 實作方向建議 |
|------|------|----------------|
| **能力雷達圖 (Skill Radar Chart)** | 讓廠商一眼看出該生的強項（例如：實作強、理論弱） | 將課程歸類（例如：資料庫類、程式語言類、系統分析類）；依據匯入的成績，在履歷上自動生成一張「能力雷達圖」，各軸為課程類別或能力維度，數值由該類成績換算（如平均分或等級）。可顯示於履歷預覽、產出 PDF/Word 或廠商審閱頁。 |
| **成績趨勢圖** | 展現學生的進步趨勢，對廠商來說是亮點 | 列出大一到大三（或各學期）的成績折線圖：X 軸為學期/學年，Y 軸為成績（如 GPA 或核心科目平均）；資料來源為已匯入的 `course_grades` 依學期彙整。可放在履歷「學業表現」區塊或獨立圖表區。 |

#### 5.5.2 AI 關鍵字萃取 (AI Extraction)

| 項目 | 說明 | 實作方向建議 |
|------|------|----------------|
| **自動標籤 (Auto-Tagging)** | 依高分科目自動產生「擅長工具」等標籤 | 若 Excel 中某科目成績達門檻（例如 85 分以上），由 AI 或規則在履歷的「擅長工具」欄位自動加上對應標籤（例如：修過「Python 程式設計」且高分 → 自動 Tag「Python」）。需維護「課程名稱 → 標籤」對照表或由 AI 從課程名稱萃取關鍵字；學生可檢視並手動增刪標籤。 |

#### 5.5.3 AI 履歷關鍵字搜尋

| 項目 | 說明 | 實作方向建議 |
|------|------|----------------|
| **依證照與高分科目撰寫／搜尋** | 讓履歷內容與證照、必修高分項目一致，並支援關鍵字搜尋 | 可根據表單的**證照資料**、**必修科目高分項目**來撰寫或建議履歷內容（例如：自傳、專長描述）；廠商或審核端可依「證照名稱」「高分課程標籤」等關鍵字搜尋學生履歷，快速篩選符合條件的應徵者。實作上可結合既有 AI 潤飾 API，輸入「證照 + 高分科目」作為 context，產出段落；搜尋則對履歷文字與標籤建立索引或簡單 LIKE 查詢。 |

### 5.6 小結與優先順序建議

- **高優先（影響資料正確性與流程收尾）**  
  學期切換前檢查、錄取時帶入志願序、重複錄取檢查、志願序截止後鎖定、學號對應檢查與匯入錯誤回報。
- **中優先（減輕人工負擔）**  
  必修 vs 已修比對、匯入前預覽與覆蓋確認、審核備註與批次操作、媒合結果匯出、切換前摘要與禁止刪除有關聯學期；**版本快照**（履歷可還原到指定版本）。
- **低優先（體驗與維運）**  
  範本與欄位說明、成績格式驗證、自動切換日誌、通知分類與已讀追蹤、公告與學期對應。
- **履歷進階（視覺化與 AI，見 5.5）**  
  能力雷達圖、成績趨勢圖、AI 自動標籤、AI 履歷關鍵字搜尋／撰寫建議，可依資源與需求排入後續迭代。

## 六、後續可延伸方向（建議）

以下為較大範圍的延伸發想，可與「五、現有功能細部完善」並行或之後再做：

1. **履歷**：履歷夾完整 CRUD、分享連結、多版比較；缺曠報表匯出、請假流程、與學期行事曆整合。
2. **志願與錄取**：志願序批次匯入、模擬分發、錄取通知信範本與排程；實習生管理：打卡、週誌、考核。
3. **公司與廠商**：公司評分、實習滿意度問卷、職缺瀏覽次數與統計；廠商端報表：應徵統計、錄取率。

6. **報表與統計**：主任/TA 儀表板圖表、年度比較、匯出 PDF/Excel 範本統一。
7. **AI**：更多語氣與情境、一鍵套用至欄位、面試題建議。
8. **系統**：備份還原、簡單健康檢查 API。
